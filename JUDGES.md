## Problem Statement

Most people have ideas for apps but no way to translate them into something a developer or AI can actually build. The gap between "I have an idea" and "here's a spec" is where most non-technical founders get stuck — and where most AI tools fail them by speaking in jargon they don't understand.

---

## Our Solution

### What We Built

**PromptReady** — an AI-powered app planning assistant that turns a raw idea into a complete, structured specification through a guided conversation. No technical knowledge required.

A user describes their app idea in plain English. Claude (Opus 4.6) generates a personalized 7-category questionnaire — Problem, Features, Design, Auth, Data, Integrations, and Logic — phrased entirely in non-technical language, with suggestion chips generated by Haiku 4.5. The user answers at their own pace, with clickable chips and on-demand explanations. Every answer is persisted to a database.

When all 7 sections are complete, a dedicated Opus pass validates the spec for completeness and surfaces any gaps or contradictions. If issues are found, the user sees them before generating — and can fix or proceed. Then a final Opus pass synthesizes all answers into a complete build artifact: a copy-paste-ready AI coding prompt, feature list, tech stack, user stories, build traps, and a phase-by-phase scope split.

The result: a structured, machine-readable product specification that any developer or AI coding tool can act on immediately.

### Why It Matters (Impact — 40%)

The addressable problem is enormous. There are an estimated 500M+ people globally with app ideas they cannot execute because they lack technical vocabulary to communicate with developers or AI tools. Current solutions either require technical literacy (GitHub Copilot, Cursor) or produce generic output (ChatGPT without structure).

PromptReady bridges this gap with three measurable outcomes:

1. **Speed** — A complete app specification in under 15 minutes, versus days or weeks with traditional product discovery processes
2. **Quality** — Opus 4.6 catches contradictions and thin answers before they become expensive misunderstandings in development
3. **Accessibility** — Zero technical jargon. Every question is translated to plain English with real-world analogies. A coffee shop owner can spec their loyalty app without knowing what a database is.

Real-world deployment scenarios: indie founders before hiring a developer, product managers before sprint planning, entrepreneurs before pitching to technical co-founders.

### How It Works (Feasibility — 15%)

The architecture is intentionally lean and deployable in minutes:

```
User → React Frontend (Vite)
         ↓
     Supabase Auth   →   Supabase DB (sessions, questionnaire_responses, build_plans)
         ↓
     Node.js API Routes (Vercel Serverless)
         ↓
     Anthropic Claude API
       ├── Opus 4.6    → questions, validation, output generation
       ├── Sonnet 4.6  → category explanations (Auth, Logic, etc.) + feature extraction
       └── Haiku 4.5   → suggestion chips, simple explanations (Design, Data)
```

**Deployment:** Single `git push` to main → Vercel auto-deploys. Public URL, no-auth browsing supported. Full auth flow (email + password with strength validation) for saving and resuming sessions.

**Reproducibility:** All dependencies pinned in `package.json`. Environment variables documented. Setup from clone to running local dev: under 5 minutes.

**Data persistence:** Supabase Postgres with Row Level Security — every user's responses are isolated, deletable, and resumable across sessions. Generated build plans are cached so users never pay the generation cost twice.

### What Makes It Different (Innovation — 15%)

**1. Four-model tiered routing by task complexity**
Most AI apps use one model for everything. We designed deliberate model routing across four distinct task types: Opus 4.6 for question generation, validation, and output generation where reasoning quality is critical; Sonnet 4.6 for nuanced explanations and structured extraction; Haiku 4.5 for suggestion chips and fast real-time responses. Each task gets the right model, not the most expensive one.

**2. Two-pass questionnaire generation**
The questionnaire is generated in two sequential passes: Opus writes the questions (reasoning-heavy — personalized, non-jargon, consultant-style), then Haiku generates the suggestion chips in a single batched call (speed-optimized — chips don't need Opus-level reasoning). This separates cognitive tasks by model strength.

**3. Pre-generation validation pass**
After all 7 steps, a dedicated Opus pass reviews the entire answer set for gaps, contradictions, and thin responses. The user sees specific feedback ("You haven't described how points expire") before the final prompt is generated — not after. This "review before you build" step is something no mainstream product discovery tool offers.

**4. Build traps + phase splits in the output**
The generated prompt doesn't just describe the app — it warns the developer about non-obvious build risks (e.g. "RevenueCat requires App Store product IDs configured before IAP works") and splits complex apps into Phase 1 (core loop) and Phase 2 (enhancements). This directly prevents the most common ways AI coding tools go wrong mid-build.

**5. Non-technical language as a first-class architectural constraint**
The system prompt enforces a strict no-jargon rule with explicit translation mappings (e.g. "Auth" → "User Accounts", "Data" → "Information"). This is architecturally enforced, not just hoped for — the model is given a list of banned words and required translations at the prompt level.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│                  React Frontend (Vite)                   │
│  LoginScreen → Intake → Questionnaire (7 steps)          │
│             → Validate → Gaps Screen → Result Screen     │
└────────────────────────┬────────────────────────────────┘
                         │ /api/*
          ┌──────────────┼──────────────┬──────────────┐
          ▼              ▼              ▼              ▼
   questionnaire.js  explain.js   validate.js    extract.js
   Opus (questions)  phase-based  Opus 4.6       Sonnet 4.6
   Haiku (chips)                                     │
                                                 generate.js
                                                 Opus 4.6
          │              │              │              │
          └──────────────┴──────────────┴──────────────┘
                         │
                  Anthropic Claude API
                         │
                  Supabase (Auth + DB)
                  sessions | questionnaire_responses | build_plans
```

## Design Decisions Log

| # | Decision | Rationale | Score Impact |
|---|----------|-----------|--------------|
| 1 | Opus 4.6 for question generation | Questions determine the quality of every downstream answer — this is the highest-leverage Opus use | Impact |
| 2 | Haiku 4.5 for suggestion chips | Chips are pattern-matching, not reasoning — Haiku is 10x faster and equally good here | Innovation |
| 3 | Opus 4.6 for pre-generation validation | Catches gaps and contradictions before they reach the user's dashboard | Impact + Innovation |
| 4 | Sonnet 4.6 for feature extraction | Structured JSON extraction from prose — needs reasoning but not Opus-level | Innovation |
| 5 | Opus 4.6 for final generation with build traps + phases | The output is the product — no compromise on quality | Impact |
| 6 | No technical jargon — architecturally enforced | Translates to genuine accessibility, not just marketing copy | Impact |
| 7 | Supabase RLS for response isolation | Users can only see their own data — security by default | Feasibility |
| 8 | Build plan caching in Supabase | Users never pay the Opus generation cost twice — resumable across sessions | Feasibility |

## Security Posture

- **No secrets in repository** — enforced by pre-commit hook scanning 15+ secret patterns
- **No files >100MB** — enforced by The Bouncer (Vibeathon compliance)
- **Row Level Security** — Supabase RLS ensures each user can only access their own data
- **Server-side API keys** — Anthropic key never touches the browser; all AI calls are proxied
- **Input validation** — all API endpoints validate required fields and types before processing
- **Auth with password rules** — minimum 8 chars, uppercase, number enforced client-side
